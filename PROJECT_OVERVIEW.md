# Decifra.ai — Project Overview

> Generated by reading the full repository (148 files, excluding `node_modules`/`.git`). No source files were modified in producing this document.

## 1. What the application does

**Decifra.ai** is a Hebrew-first (RTL) SaaS web app that helps consumers understand legal/financial documents (employment contracts, rental agreements, pay slips, insurance policies, loan agreements, purchase agreements). The core loop is:

1. User uploads a document (PDF/image).
2. An LLM (via the Base44 platform's `InvokeLLM` integration) extracts structured data and produces a plain-language analysis: executive summary, key legal terms, risks, obligations, financial terms, inconsistencies, cross-references, etc. Pay slips get a specialized "compliance check" prompt (minimum wage, deduction sanity checks).
3. The user browses the analysis in a tabbed UI, gated by subscription plan (free / beginner / pro / business).
4. For an extra one-time fee, the user can request a **human lawyer review** — the document + AI summary is routed to a lawyer dashboard, and the lawyer submits a written legal opinion back to the client.
5. Subscription upgrades and lawyer-review purchases are paid for via **Stripe Checkout**, orchestrated through Base44 serverless functions.

It's built on **Base44**, a low-code/AI app platform that provides authentication, a hosted entity/database layer with declarative Row-Level-Security (RLS), file storage, LLM integrations, and Deno-based serverless "functions" — the repo is effectively a Base44 app plus a standard Vite/React frontend on top of it.

## 2. Technologies used

- **Frontend**: React 18, React Router v6, Vite 6, Tailwind CSS + shadcn/ui (Radix UI primitives, `class-variance-authority`, `tailwind-merge`), Framer Motion, `@tanstack/react-query`, `react-hook-form` + `zod`, `lucide-react` icons, `react-markdown`, `jspdf`/`html2canvas` (PDF export), `recharts`, `sonner`/`react-hot-toast` (toasts).
- **Platform/backend**: `@base44/sdk` (auth, entities/database, file storage, LLM/email/SMS integrations, service-role access), `@base44/vite-plugin`. Backend "functions" are Deno serverless handlers (`Deno.serve`) living under `base44/functions/*/entry.ts`, using `npm:@base44/sdk` and `npm:stripe` imports (Deno's npm compat layer).
- **Payments**: Stripe (`stripe` npm package server-side, `@stripe/stripe-js` / `@stripe/react-stripe-js` client-side, though Checkout redirect is used rather than embedded Elements).
- **Language/tooling**: JavaScript/JSX (not TypeScript, despite `tsconfig`-style `jsconfig.json` and a `typecheck` script), ESLint 9 flat config, PostCSS/Autoprefixer.
- **Entities/config-as-code**: `base44/entities/*.jsonc` (JSON Schema + RLS rules), `base44/config.jsonc`, `base44/.app.jsonc` (app ID).

## 4. Authentication flow

1. **`AuthProvider`** (`src/lib/AuthContext.jsx`) runs on app mount. It first calls the Base44 **public-settings** endpoint (`/api/apps/public/prod/public-settings/by-id/{appId}`) with whatever token is present, to determine whether the app requires auth and whether the current user is registered for it. This can produce `auth_required` or `user_not_registered` error states, handled in `App.jsx` (redirect to login, or render `UserNotRegisteredError`).
2. If a token is present, `base44.auth.me()` is called to fetch the authenticated user and flip `isAuthenticated`.
3. The token itself comes from **`src/lib/app-params.js`**, which reads `access_token` (and `app_id`, `server_url`) from the URL query string (stripping it from the URL after reading) or from `localStorage`, falling back to Vite env vars (`VITE_BASE44_APP_ID`, `VITE_BASE44_BACKEND_URL`). This is how the app receives its session when embedded/redirected by the Base44 platform's own login flow (`base44.auth.redirectToLogin()` / `base44.auth.logout()`).
4. Once authenticated, most pages call **`getCurrentUser()`** (`src/components/userSync.jsx`) rather than raw `base44.auth.me()`. This function:
   - Builds a "base" user object from the auth session (id, email, name, a client-computed `role` that special-cases the hardcoded email `decifratech@gmail.com` as `"admin"`).
   - Calls the **`ensureUserProfile`** Base44 function (server-side, uses `asServiceRole`) to find-or-create a row in the custom **`Users`** entity keyed by `authId`, optionally emailing an admin notification on first signup.
   - Merges the DB `Users` row over the base auth object (DB `role`/`plan` win, except the hardcoded admin email always forces `role: "admin"`, `plan: "business"`).
   - Falls back to a read-only `Users.filter({authId})` if the function call fails, and finally to an auth-only object if even that fails — designed to never return `null` once a session exists.
5. `Layout.jsx` re-runs `getCurrentUser()` on mount and on `window focus` (to pick up state after an OAuth redirect back into the tab).

## 5. AI analysis flow

1. **Upload** (`src/pages/Upload.jsx`): user picks a category → document type → file. Before uploading, it force-refreshes the current user and enforces a **client-side scan quota** by counting `Document.filter({ ownerAuthId })` against a hardcoded `planLimits` map (`free: 1, beginner: 10, pro: 30, business: 100`), skipped entirely for admins/dev-bypass users.
2. The file is uploaded via `UploadFile` (Base44 Core integration) to get a `file_url`.
3. A large Hebrew prompt (different for `pay_slip` vs. everything else) plus a matching `response_json_schema` is sent to **`InvokeLLM`** with `file_urls: [file_url]`, requesting a structured JSON object (executive summary, key points, risks, obligations, financial terms, inconsistencies, cross-references, or for pay slips: salary breakdown + compliance check).
4. A new **`Document`** row is created with both `analysisResult` (new field) and `ai_analysis` (legacy field, duplicated for backward compatibility) plus `analysisStatus: "done"`.
5. The user is redirected to **`Analysis.jsx?id=...`**, which reloads the document, migrates legacy `ai_analysis`-only documents into `analysisResult` on the fly, and can (re-)run `analyzeDocument()` if no result exists yet (guarded by a `hasStartedRef` to avoid duplicate runs, plus a "re-analyze" button gated to `pro`+ plans).
6. The analysis UI builds its tab list dynamically based on which fields are non-empty, and layers additional plan-gated widgets on top: `ImpactCards`, `AIAdvisor`, `CompareDocuments`, `LegalResearch`, `ExportMenu`, a "reality mode" simulation of how the contract behaves under stress scenarios, notes, and a document chatbot — most of these call `InvokeLLM` again independently and are locked/unlocked by `currentPlan` computed client-side (see §8).

## 6. Lawyer review flow

1. From `Analysis.jsx`, a "Send to lawyer" button (disabled on the `free` plan, or if already requested/completed) opens **`LawyerReviewModal`**, which shows plan-based pricing and, on confirmation, calls the **`createLawyerReviewPayment`** Base44 function with the `documentId`.
2. That function looks up the user's plan from the **`Users`** entity, computes a discounted price (`BASE_PRICE = 200 ILS`, discounts 0/10/20/30% for free/beginner/pro/business), creates a **Stripe Checkout session** with `documentId`/`userId`/`plan` in `metadata`, and returns the checkout URL.
3. On success, Stripe redirects to `/lawyer-dashboard?session_id=...&document_id=...`. **`completeLawyerReviewPayment`** re-verifies the session with Stripe (`payment_status === 'paid'`), then:
   - Creates a **`LawyerCase`** row (`documentId`, `clientUserId`, `client_name`, `client_email`, `client_summary` pulled from the document's `ai_analysis.executive_summary`, `status: "pending"`).
   - Updates the `Document.status` to `"lawyer_requested"`.
   - Emails every user with `role: "lawyer"` in the `Users` entity.
4. **`LawyerDashboard.jsx`** is a role-gated page (redirects anyone whose `role` isn't `lawyer`/`admin`) showing pending/in-progress/completed `LawyerCase` records, cross-referenced with `Document.list()`. Lawyers can write a free-text opinion (with canned templates per document type), attach files, and save private internal notes.
5. **`submitReview()`** marks the `LawyerCase` completed, writes a `lawyer_review` object onto the `Document` (summary, recommendation, signed date, lawyer name) and sets `Document.status = "completed"`, creates a `Notification` for the client (correct field names — see §8 for the *other*, inconsistent notification helper), and emails the client.
6. Back in `Analysis.jsx`, once `document.status === 'completed'` or `document.lawyer_review` is present, a "Lawyer's Opinion" card is rendered inline with the AI analysis.

## 7. Payment flow

Two independent Stripe Checkout flows, both server-side session creation + client-side redirect + a "complete" endpoint hit after redirect back:

**A. Plan upgrade** (`Upgrade.jsx` → `createUpgradePayment` → Stripe → `CompleteUpgradePayment.jsx` → `completeUpgradePayment`):
- Prices are hardcoded in the function (`beginner`/`pro`/`business` × monthly/yearly, in ILS).
- Session `metadata` records `userId`, `userAuthId`, `plan`, `billingPeriod`.
- On return, `completeUpgradePayment` verifies `session.payment_status === 'paid'`, looks up the `Users` row by the *authenticated caller's* `authId`, and (using `asServiceRole`) sets `plan` + resets `scansUsed: 0`.
- **However**, the `plan` value it writes comes from the **URL query string** (`?plan=...`), not from the Stripe session's own `metadata.plan` — see §8.

**B. Lawyer review** (`LawyerReviewModal`/`LawyerReviewPayment` → `createLawyerReviewPayment` → Stripe → `completeLawyerReviewPayment`, described in §6): price is computed server-side from the user's stored plan at checkout-creation time, so this leg is sounder than the upgrade flow, but the *completion* endpoint still trusts a client-supplied `documentId` without cross-checking it against `session.metadata.documentId` (§8).

Both "complete" endpoints are idempotent-unsafe: nothing marks a Stripe session as "already consumed," so a captured `session_id` could in principle be replayed against the completion endpoint more than once.

## 8. Database entities

All entities are defined declaratively in `base44/entities/*.jsonc` with JSON-Schema-style `properties` and a Base44 RLS block (`{{user.id}}` / `{{user.email}}` / `user_condition.role` templates enforced server-side).

| Entity | Key fields | RLS summary |
|---|---|---|
| **`Document`** | `ownerAuthId`, `userId` (legacy dup), `folderId`, `title`, `category`, `documentType`/`document_type` (new/legacy dup), `file_url`, `ai_analysis` (legacy result blob), `analysisResult`/`analysisStatus`/`analyzedAt` (new fields), `status` (uploaded→analyzing→analyzed→lawyer_requested→lawyer_reviewing→completed), `lawyer_review`, `user_notes`, `isFavorite`, `tags` | read/update/delete: owner (`ownerAuthId` **or** `created_by` email) |
| **`Folder`** | `name`, `ownerAuthId`, `color`, `icon` | all owner-only via `ownerAuthId` |
| **`LawyerCase`** | `documentId`, `clientUserId`, `lawyerUserId`, `client_name`, `client_email`, `client_summary`, `urgency`, `estimated_hours`, `status` (pending/in_progress/completed), `attachments[]`, `lawyer_notes` | create: client only; read: client, assigned lawyer, or admin; update: lawyer or admin; delete: admin |
| **`Notification`** | `userId` (stores an **email**, despite the name/description), `title`, `message`, `type` (document_reviewed/new_message/system), `documentId`, `isRead`, `createdAt` | all ops keyed on `userId == {{user.email}}`, or admin |
| **`User`** | `plan`, `scansUsed` | extension fields bolted onto Base44's *built-in* auth user object |
| **`Users`** | `authId`, `email`, `name`, `role` (user/lawyer/admin), `plan`, `scansUsed`, `alertSettings{highRiskAlert, analysisCompleteAlert, lawyerResponseAlert, weeklyDigest}` | create/update: admin only (app-level creation goes through the `ensureUserProfile` service-role function); read: self or admin |

Two structurally separate "user" concepts coexist (`User` — the built-in auth entity with an ad-hoc `plan`/`scansUsed` extension — and `Users` — a fully custom profile entity with `role`, richer settings, etc.). The app's actual runtime logic is built almost entirely around `Users` (via `userSync.jsx`); the `User.jsonc` extension appears to be a mostly vestigial/parallel definition. See §9.

## 9. Potential architectural issues

Ordered roughly by severity/impact:

1. **Client-controlled plan on subscription completion.** `completeUpgradePayment` (`base44/functions/completeUpgradePayment/entry.ts:17-42`) takes `plan` from the **URL query string** the browser sends, not from the Stripe session's own `metadata.plan` that was set at checkout-creation time. Since the Stripe session is only checked for `payment_status === 'paid'` (not for which plan/amount was actually paid), a user could pay for the cheapest plan and then hit the completion endpoint with `?plan=business` to be upgraded to Business for the Beginner price. The fix is straightforward (trust `session.metadata.plan`, not the query string) but as written this is a real privilege/pricing bypass.
2. **Plan gating on the Analysis page can be overridden via URL parameter.** In `Analysis.jsx` (~line 90-95), `currentPlan` is computed as `paramPlan || userPlanLocal`, where `paramPlan` comes straight from `?plan=` in the URL. Every feature gate on that page (`canAccess(currentPlan, ...)`, export, reanalyze, legal research, lawyer-review button) is driven by this value, so simply adding `?plan=business` to any Analysis URL unlocks every paid feature client-side. This appears to be leftover test/dev scaffolding that shipped into the real gating logic.
3. **`LawyerCase.documentId` vs. `document_id` field mismatch.** The entity schema (and the record actually written by `completeLawyerReviewPayment`) uses `documentId`, but almost all of `LawyerDashboard.jsx`'s reads use `caseItem.document_id` (snake_case) — e.g. `getCasesByStatus`/filters/sort/case-detail rendering. Only one spot (the "new case" notification list) correctly uses `caseItem.documentId`. In practice this likely means the lawyer dashboard's document lookups (`getDocumentById(selectedCase.document_id)`) silently return `undefined` almost everywhere, breaking the "document + client info" and "full AI analysis" panels for lawyers.
4. **`sendNotification()` writes fields that don't match the `Notification` schema/RLS**, and is dead code besides. `notificationService.jsx` creates notifications with `user_email`/`document_id`/`is_read`/`sent_email`, while the entity schema (and every other call site, e.g. `LawyerDashboard.submitReview`) uses `userId`/`documentId`/`isRead`/`createdAt`. Since the entity's RLS `create` rule requires `userId == {{user.email}}`, a `sendNotification()` call would likely fail RLS or create an orphaned row nobody can read. It's imported in `Analysis.jsx` and `LawyerDashboard.jsx` but never actually invoked anywhere — apparently abandoned mid-refactor.
5. **`NotificationSettings.jsx` targets a field the schema doesn't have.** It reads/writes `user.notificationSettings` via `base44.auth.updateMe(...)`, but `Users.jsonc` defines `alertSettings` with a different shape entirely. This looks like an orphaned feature that never got reconciled with the entity schema — user-configured notification preferences are effectively not persisted anywhere the rest of the app can read.
6. **Dual "user" entities (`User` vs `Users`)** create ongoing confusion risk: `User.jsonc` extends Base44's built-in auth user with `plan`/`scansUsed`, while nearly all real logic (`userSync.jsx`, all the Stripe functions, `LawyerDashboard`, `UserManagement`) reads/writes the separate custom `Users` entity instead. It's unclear which one is authoritative for `plan`/`scansUsed`, and nothing appears to keep them in sync.
7. **Admin/role/plan checks are enforced almost exclusively client-side.** `isAdminUser`/`canBypassPlan` (`devAccess.jsx`), the `role !== 'admin'` redirect guards in `UserManagement.jsx`/`LawyerDashboard.jsx`, and every `FeatureGate`/`AdvancedToolsGrid`/`ExportMenu` lock are plain UI conditionals. The real backstop is entity RLS (server-side) for reads/writes, but several privileged actions go through `base44.asServiceRole` calls invoked directly from browser code (e.g. `UserManagement.jsx`'s role/plan editing, `ensureUserProfile`'s user creation) — the security boundary there depends entirely on how Base44 authorizes `asServiceRole` calls from a given caller, which isn't visible in this repo. A hardcoded admin email (`decifratech@gmail.com`) is also baked into `userSync.jsx`'s client-side fallback role resolution.
8. **No idempotency/replay protection on payment completion endpoints.** Neither `completeUpgradePayment` nor `completeLawyerReviewPayment` records that a given Stripe `session_id` has already been consumed, nor does the lawyer-review completion endpoint verify the client-supplied `documentId` against `session.metadata.documentId`. A captured success-URL could be replayed (re-granting a plan reset, or attaching one paid session to a different document's `LawyerCase`).
9. **Unmetered/ungated AI cost surfaces.** `Support.jsx`'s public chatbot and several Analysis widgets (`ImpactCards`, `AIAdvisor`, `CompareDocuments`, `LegalResearch`, `DocumentChatbot`) call `InvokeLLM` on mount or on user action with little-to-no plan/quota check *inside* the component itself — gating, where it exists, is only about what's *displayed*, not whether the (costly) LLM call happens at all.
10. **Dead code / cruft**: empty, unrouted `Dashboard.jsx`/`Documents.jsx`/`CRM.jsx`; near-duplicate `RealityMode.jsx` and `SimulationEngine.jsx` implementing the same feature twice; `UserManagement.jsx`'s "Edit Quota" dialog reads/writes a `scanQuota` field that doesn't exist in the `Users` schema (only `scansUsed` does); `ThemeProvider.jsx` has dark mode fully disabled as a no-op yet `Settings.jsx` still renders a working-looking dark/light toggle; `jsconfig.json`/`eslint.config.js` scope excludes large parts of `src` (e.g. `src/lib`, `src/components/ui`) from linting/type-checking coverage.
11. **Legacy/duplicate fields throughout `Document`** (`document_type` vs `documentType`, `ai_analysis` vs `analysisResult`, `userId` vs `ownerAuthId`) are kept in sync manually at write time in application code rather than derived — every new write path has to remember to populate both, and at least one downstream consumer (`ExportMenu.jsx`, per the sub-agent survey) only reads the legacy field, so PDF exports can be blank for documents that only populated the new field.

## 10. Questions I still have

- **Is `base44.asServiceRole` actually restricted server-side** to specific trusted call sites, or can any authenticated browser session invoke it for arbitrary entities/updates? This matters a lot for how serious issue #7 (client-invoked service-role calls for admin actions) really is — I couldn't find that enforcement in this repo since it lives in the Base44 platform, not this codebase.
- **Is the `?plan=` override in `Analysis.jsx` (#2) intentional dev/QA tooling** (parallel to the explicit `devAccess.jsx` dev-mode flag) that was meant to be removed before production, or is it live in the deployed app today? If live, this seems like the highest-priority fix.
- **Which of `User` vs `Users` is meant to be authoritative** for `plan`/`scansUsed` going forward — is `User.jsonc` a remnant of an earlier design that should be deleted, or is there a sync mechanism elsewhere (e.g., a Base44-platform-level hook) I don't have visibility into?
- **Are `RealityMode.jsx` and `SimulationEngine.jsx` both still wired up in the live UI**, or is one an abandoned in-progress replacement for the other?
- **Is there a webhook-based Stripe integration anywhere** (e.g., a `stripe.webhooks` handler) that I missed, which would make the client-driven "complete payment" endpoints a secondary/defense-in-depth path rather than the *only* mechanism confirming payment? I did not find one in `base44/functions/`.
- **What actually happens to a `LawyerCase`/`Document` if `completeLawyerReviewPayment`'s `documentId`/session mismatch (#8) is exploited** — is there any downstream reconciliation (e.g., a nightly job against Stripe's own records) that would catch it, given this repo has no visible admin reporting on payments vs. entities created?
- **Where is `alertSettings` (the schema-defined notification preferences) actually read and acted upon**, if anywhere? I didn't find a consumer in the pages/components surveyed, which would mean the entire notification-preferences feature (both the schema field and the mismatched `notificationSettings` UI) is currently inert.
